# 32. 日志与监控基础：出了问题，要能看见问题

## 学习目标

- 理解日志和监控为什么是排查问题的基础。
- 知道“记录什么”比“记录多少”更重要。
- 建立对关键路径埋点和错误观察的基础认知。

## 场景引入

很多新手遇到 bug 时，最痛苦的不是 bug 本身，而是根本不知道问题发生在哪里。比如：

- 用户说页面打不开，但你本地复现不了
- 某个按钮点击后没结果，但没有报错提示
- 某个异步流程中间失败了，却不知道卡在哪一步

这时候日志和监控就很重要。它们的作用是帮你“看见问题”。

## 本章术语先看懂

- `日志`：记录程序运行过程中的关键事件。
- `监控`：持续观察应用运行状态和异常情况。
- `关键路径`：对用户最重要、最容易出问题的操作流程。
- `错误上报`：把发生的错误收集起来，方便后续分析。

## 一句话理解

没有日志和监控，很多问题就像在黑盒里发生，你几乎无从下手。

## 一个基础日志示例

```swift
import SwiftUI

struct LoggingDemoView: View {
    @State private var status = "等待操作"

    var body: some View {
        VStack(spacing: 12) {
            Text(status)

            Button("开始同步") {
                log("用户点击了开始同步按钮")
                status = "同步中..."

                Task {
                    await fakeSync()
                }
            }
            .buttonStyle(.borderedProminent)
        }
        .padding()
    }

    private func log(_ message: String) {
        print("[SyncFlow] \(message)")
    }

    @MainActor
    private func fakeSync() async {
        log("同步任务开始")
        try? await Task.sleep(for: .seconds(1))
        status = "同步完成"
        log("同步任务完成")
    }
}

#Preview {
    LoggingDemoView()
}
```

## 代码拆解（小白重点）

- 日志不只是“打印一下”，而是帮助你还原用户操作和程序过程。
- 给日志加上简单前缀，比如 `[SyncFlow]`，后面排查时会更容易找。
- 最值得记录的是关键节点：开始了、成功了、失败了、用户点了什么。

## 记录日志时该注意什么

- 记录关键动作，不要什么都打。
- 不要把密码、token 这类敏感内容写进日志。
- 日志文案尽量能说明上下文，而不是只写“到了这里”。

## 新手常见误区

- 出问题才想起加日志。
- 调试时什么都打印，真正排查时反而看不清。
- 日志里出现敏感数据。

## 新手排错流程

1. 先给关键流程补开始、成功、失败三个节点日志。
2. 发现日志很多但没帮助时，检查是不是没有上下文信息。
3. 排查线上问题时，优先关注用户关键路径相关日志。

## 章节小测（带答案）

### 题 1

日志最核心的价值是什么？

参考答案：帮助你还原程序运行过程，定位问题发生在哪一步。

### 题 2

为什么不应该什么都打印？

参考答案：因为噪音太多会让真正有价值的信息更难看见。

### 题 3

为什么日志里不能随便放敏感信息？

参考答案：因为日志也可能被查看、上传或泄露，暴露敏感信息会带来风险。

## 练习任务

- 基础练习：给你的小项目补 3 个关键节点日志。
- 加强练习：为一个失败场景增加更清晰的错误记录。
- 挑战练习：整理一份“哪些信息可以记日志，哪些不可以”的清单。

## 本章小结

学完这章后，你应该已经知道：日志和监控不是锦上添花，而是帮你发现问题、定位问题的基础能力。
