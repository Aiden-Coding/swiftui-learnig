# 00. 常见报错与排错清单：新手最容易卡住的 SwiftUI 问题

## 为什么需要这份清单

很多新手学 SwiftUI 的时候，不是卡在“完全不会”，而是卡在这些高频瞬间：

- 一敲代码就报错
- 页面跑不起来
- 明明写得很像教程，结果效果不对
- 一个小错误能卡半小时

这份清单就是为了帮你先绕开最常见的坑。

## 先记住最重要的排错原则

### 原则 1：先看第一条报错

很多报错是连锁反应。

真正的根因往往在第一条，而不是最后一条。

### 原则 2：一次只改一两处

不要因为着急，把一大段一起改掉。这样你更难知道问题到底是哪里造成的。

### 原则 3：先回到最小可运行版本

如果你越改越乱，就先删回最基础版本，让页面先重新跑起来，再一点点加回功能。

## 高频问题 1：`Cannot find ... in scope`

### 常见原因

- 变量名拼错
- 视图名拼错
- 某个值定义在别的作用域里，你这里访问不到

### 怎么排查

1. 检查大小写是否一致
2. 检查变量是不是在当前结构体或函数里定义的
3. 检查是不是写错了前缀或少传了参数

## 高频问题 2：`Extra argument in call`

### 常见原因

- 你传了一个当前组件并不接受的参数
- 参数标签写错
- 多写了一个逗号或括号结构乱了

### 怎么排查

1. 对照当前组件定义，确认它到底需要哪些参数
2. 检查参数标签名是否和定义一致
3. 看最近一两行括号是否配对

## 高频问题 3：`Missing argument for parameter`

### 常见原因

- 某个必须传入的参数没传
- 调用视图或函数时漏掉了一项

### 怎么排查

1. 点进报错对应的初始化方法或视图定义
2. 看看哪些参数没有默认值
3. 把缺少的那一项补上

## 高频问题 4：界面不刷新

### 常见原因

- 你改的不是 `@State`
- 子视图没用 `@Binding`
- 共享对象里的属性没加 `@Published`

### 怎么排查

1. 先问自己：这个数据是谁拥有的
2. 当前值是不是放在 `@State` 里
3. 如果是子视图修改父视图数据，是否传了绑定
4. 如果是共享对象，属性是否标记成了 `@Published`

## 高频问题 5：按钮点了没反应

### 常见原因

- 按钮 action 没有真正修改状态
- 修改后 UI 没绑定那份状态
- 按钮层级或点击区域有问题

### 怎么排查

1. 在按钮点击里先加一句简单输出，确认是否真的触发
2. 检查按钮点击后改的是不是正确的状态变量
3. 检查界面是不是读取了同一份状态

## 高频问题 6：列表显示不出来

### 常见原因

- 数据数组为空
- `ForEach` 识别不到唯一标识
- 数据没有真正传到列表视图

### 怎么排查

1. 先看数据源里到底有没有内容
2. 检查 `ForEach` 是否有 `id` 或模型是否遵守 `Identifiable`
3. 确认当前列表读到的是不是你以为的那份数据

## 高频问题 7：导航不跳转

### 常见原因

- 没有放在 `NavigationStack` 里
- `NavigationLink` 的目标视图写错
- 跳转逻辑依赖的状态没有变化

### 怎么排查

1. 确认外层有没有 `NavigationStack`
2. 检查目标视图是否可正常初始化
3. 如果是状态驱动导航，检查状态是否真的改变

## 高频问题 8：表单输入异常

### 常见原因

- `TextField` 没绑定到状态变量
- 状态类型和控件不匹配
- 提交后没有做清空或校验逻辑

### 怎么排查

1. 检查控件是不是绑定到了 `$变量`
2. 检查变量类型是否正确
3. 检查按钮逻辑是否处理了空输入和边界情况

## 高频问题 9：异步页面一直转圈

### 常见原因

- 加载状态没有从 `.loading` 切换出去
- 异步函数没有走到成功或失败分支
- 按钮重试时没有重新启动异步任务

### 怎么排查

1. 先确认异步函数是否真的执行到了末尾
2. 检查成功和失败状态是否都有覆盖
3. 检查重试逻辑是不是重新触发了 `Task`

## 高频问题 10：Preview 跑不起来

### 常见原因

- 某些依赖环境没有提供
- SwiftData / 共享对象没有注入
- 当前代码本身有编译错误

### 怎么排查

1. 先确认项目整体能不能编译
2. 如果用了环境对象或数据容器，检查 Preview 是否补齐
3. 先让视图回到最小依赖版本，验证 Preview 本身是否正常

## 遇到报错时最推荐的排错顺序

可以直接照这个顺序做：

1. 看第一条报错
2. 看报错所在行
3. 看上一行和下一行括号、逗号、引号是否配对
4. 看变量名和参数名是否拼写一致
5. 回退最近 1-2 处改动
6. 回到最小可运行版本

## 什么时候该停下来，不要继续乱改

如果你出现下面这些情况，就说明应该先停一下：

- 同一段代码连续改了很多次，越来越乱
- 新报错越来越多
- 已经忘了最开始到底想解决什么问题

这时候最好的办法通常不是继续试，而是：

- 回到上一个能运行的版本
- 重新做最小改动

## 最后的提醒

报错不是说明你不适合学 SwiftUI，而是学习过程中非常正常的一部分。

真正重要的是：

- 你有没有一套稳定的排错顺序
- 你是不是越报错越乱改

会排错，本身就是能力的一部分。
