# 39. 离线优先基础：没有网络时，应用也别完全失能

## 学习目标

- 理解为什么“没有网络也能用一点”很重要。
- 知道离线优先和单纯网络加载的思路区别。
- 建立对缓存、本地回退体验的基础认知。

## 场景引入

很多应用一断网就直接空白，或者不停转圈，用户完全不知道能不能继续用。这种体验会很差。

离线优先的思路不是“所有功能都能离线完成”，而是优先保证：

- 用户至少还能看到一些已有内容
- 关键操作知道现在不能执行还是稍后重试
- 页面不要因为没网就彻底失去反馈

## 本章术语先看懂

- `离线优先`：优先考虑无网络或弱网络情况下的可用性。
- `缓存`：把之前成功获取的数据临时保存在本地。
- `降级体验`：在条件变差时，依然提供一个可接受的备选体验。
- `同步`：网络恢复后再把本地变化和远端状态对齐。

## 一句话理解

离线优先不是让所有功能都离线可用，而是让应用在断网时不要完全瘫痪。

## 一个最基础的离线优先思路

1. 优先显示本地已有数据。
2. 后台尝试刷新最新数据。
3. 如果请求失败，保留旧内容并告诉用户当前是离线或加载失败状态。
4. 等网络恢复后再同步。

## 新手常见误区

- 一断网就直接把页面清空。
- 请求失败后既没有旧数据，也没有提示。
- 把离线优先理解成必须离线支持所有复杂功能。

## 新手排错流程

1. 先想页面在断网时最少还能展示什么。
2. 优先给失败和离线状态补清晰提示。
3. 如果已有旧数据，不要轻易在失败时把它清掉。

## 章节小测（带答案）

### 题 1

离线优先最核心的目标是什么？

参考答案：让应用在没有网络时依然保留基本可用性和清晰反馈。

### 题 2

为什么请求失败时不一定要立刻清空旧数据？

参考答案：因为旧数据通常仍比完全空白更有价值，能帮助用户继续理解当前内容。

### 题 3

离线优先是不是意味着所有功能都必须能离线完成？

参考答案：不是，它更强调在断网时尽量保留关键体验，而不是让所有能力都完整可用。

## 练习任务

- 基础练习：为一个课程列表页设计断网时的页面状态。
- 加强练习：思考哪些内容适合缓存，哪些不适合。
- 挑战练习：为“离线状态下的任务打卡”设计恢复同步思路。

## 本章小结

学完这章后，你应该已经知道：离线优先的意义，不是做复杂技术炫技，而是让用户在糟糕网络环境下也不会完全失去控制感。
