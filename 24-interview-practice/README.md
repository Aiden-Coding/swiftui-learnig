# 24. 面试与练习：把学过的知识说清楚、做出来

## 学习目标

- 学会用更清楚的语言表达自己对 SwiftUI 基础知识的理解。
- 知道面试里常见的基础问法有哪些。
- 能把“学过”和“会说会做”连接起来。

## 场景引入

很多人学了不少内容，但一到面试、分享或者自我介绍时，就会卡住：

- 学过 `@State`，但说不清楚它和 `@Binding` 的区别
- 做过列表和导航，但讲不出自己是怎么组织页面的
- 代码能写一点，但不会把思路讲明白

这一章的目的，就是帮你把已经学过的知识整理成能表达、能复述、能练习的形式。

## 本章术语先看懂

- `知识表达`：把自己理解的内容讲清楚。
- `场景题`：面试中基于真实开发问题提问。
- `手写题`：让你现场写一个小功能或解释写法。

## 一句话理解

面试不只是考你会不会 API，更在看你能不能把思路讲清楚。

## 常见基础问题示例

### 1. `@State` 和 `@Binding` 有什么区别？

参考回答思路：

- `@State` 表示当前视图自己拥有的数据。
- `@Binding` 表示子视图拿到一个可读可写的引用。
- 一个负责“拥有”，一个负责“协作修改”。

### 2. `VStack`、`HStack`、`ZStack` 分别适合什么场景？

参考回答思路：

- `VStack` 做纵向排列
- `HStack` 做横向排列
- `ZStack` 做层叠布局

### 3. 什么时候用 `sheet`，什么时候用导航？

参考回答思路：

- 导航适合进入下一层内容结构
- `sheet` 适合临时处理一件事，比如新增、编辑、确认

### 4. 为什么 SwiftUI 强调状态驱动界面？

参考回答思路：

- 因为界面会随着状态变化自动更新
- 开发者更关注“界面现在应该是什么样”，而不是手动刷新控件

## 小练习：尝试自己回答

你可以练习用自己的话回答下面几个问题：

- 什么情况下你会抽组件？
- 为什么异步页面要有加载中和失败状态？
- 你做过的小项目里，状态是怎么组织的？
- 如果页面越来越乱，你会怎么拆？

## 新手常见误区

- 只背术语，不理解场景。
- 只会说定义，不会结合例子。
- 只会写代码，不会讲设计思路。

## 新手排错流程

1. 回答问题总是很空时，先补“场景 + 原因 + 做法”。
2. 讲不清区别时，试着找一个最直观的对比案例。
3. 紧张时，先从自己做过的页面讲起，不要强行背模板。

## 章节小测（带答案）

### 题 1

为什么面试里不只考 API 名字？

参考答案：因为面试更想了解你是否真的理解使用场景和思路。

### 题 2

回答技术问题时，为什么要尽量带场景？

参考答案：因为带场景会让回答更具体，也更能体现真实理解。

### 题 3

如果你已经会写，但说不清楚，下一步最该练什么？

参考答案：练习把自己的项目经验和基础概念用简单语言复述出来。

## 练习任务

- 基础练习：把 `@State`、`@Binding`、`ObservableObject` 的区别各用一句话讲清楚。
- 加强练习：录一段 3 分钟口述，介绍你做过的一个 SwiftUI 小页面。
- 挑战练习：整理 10 个你觉得最可能被问到的基础问题，并自己写答案。

## 本章小结

学完这章后，你应该已经知道：真正的掌握，不只是会写代码，还包括能把自己的理解和设计思路讲清楚。

## 本章完成标准

学完这一章后，你至少应该能做到：

- 能用自己的话解释几个核心 SwiftUI 概念
- 能把“我会写”进一步变成“我能讲清楚”
- 能发现自己哪些知识点其实还停留在模糊状态

## 学完这一章后马上可以做的练习

挑 3 个你最熟的主题，尝试不用看教程，直接用自己的话讲给别人听，比如：

- `@State` 和 `@Binding`
- `List` 和 `ForEach`
- `sheet` 和导航的区别

只要你讲的过程中卡住了，就说明那个点还值得回去再补。
