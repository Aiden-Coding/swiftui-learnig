# 42. 代码评审基础：不是挑刺，而是一起把问题挡在上线前

## 学习目标

- 理解代码评审的真正目的。
- 知道评审时最值得优先关注什么。
- 建立“先看风险，再看风格”的评审意识。

## 场景引入

很多新手第一次看到代码评审，会以为这只是“别人来挑我写得不够好”。其实好的代码评审最核心的价值，是在问题上线前，把风险更早发现出来。

真正有价值的评审，通常会优先关注：

- 功能逻辑是否正确
- 是否有明显边界遗漏
- 是否可能引入回归问题
- 是否缺少必要测试

而不是先纠结某个空格或命名细节。

## 本章术语先看懂

- `代码评审`：在代码合入前，由他人检查实现质量和风险的过程。
- `回归问题`：修改一个地方后，导致原本正常的功能被破坏。
- `边界情况`：极端输入、空数据、失败路径等。
- `可维护性`：代码以后是否容易读、改、扩展。

## 一句话理解

好的代码评审不是为了挑刺，而是为了更早发现问题、减少线上风险。

## 评审时优先看什么

1. 功能有没有按需求真正实现。
2. 失败路径和空状态有没有处理。
3. 改动会不会影响旧功能。
4. 代码是否清楚到别人能接着维护。
5. 是否缺少关键测试或验证说明。

## 新手常见误区

- 一上来只看命名和格式。
- 不去想用户真实会怎么点、怎么输入。
- 看到“能跑”就默认没问题。

## 新手排错流程

1. 先从功能和风险角度读代码，而不是先抠风格。
2. 想想这段改动最可能在哪些情况下出问题。
3. 如果一个改动影响多个流程，优先确认是否缺测试或验证说明。

## 章节小测（带答案）

### 题 1

代码评审最核心的价值是什么？

参考答案：更早发现逻辑风险、边界遗漏和回归问题，减少问题流入线上。

### 题 2

为什么评审时不应该先盯着格式细节？

参考答案：因为真正更高优先级的是功能正确性、风险和可维护性。

### 题 3

看到一段代码“能跑”，为什么还不能立刻放心？

参考答案：因为它可能只覆盖了成功路径，仍然遗漏失败场景、边界情况或回归风险。

## 练习任务

- 基础练习：选一段自己的代码，列出 3 个可能的边界情况。
- 加强练习：为一个页面改动写一段“自我评审说明”。
- 挑战练习：练习从“功能、边界、回归、测试”四个角度检查一个小功能。

## 本章小结

学完这章后，你应该已经知道：代码评审的重点不是挑语气，而是一起把真正的问题挡在上线前。
