# 05. 状态管理基础：@State、@Binding

## 学习目标

- 理解 SwiftUI 为什么强调“状态驱动界面”。
- 学会使用 `@State` 保存当前视图的数据。
- 学会使用 `@Binding` 把可修改的数据传给子视图。
- 建立一个非常关键的判断标准：谁拥有数据，谁就负责状态。

## 场景引入

你写了一个按钮，点击后想让数字加一，结果要么界面不更新，要么子视图改不了父视图的数据。这些问题本质上都和“状态”有关。

很多新手前几章学布局和控件时还觉得顺手，一到状态就开始混乱，常见表现是：

- 变量明明改了，界面没反应
- 子视图拿到的是值，但改不动父视图
- 不知道什么时候该用 `@State`，什么时候该用 `@Binding`

如果你把这一章学扎实，后面的表单、动画、异步加载、数据流都会轻松很多。

## 本章术语先看懂

- `@State`：当前视图自己拥有、自己维护的可变状态。
- `@Binding`：对子视图开放的可读可写引用。
- `状态驱动 UI`：界面不是手动刷新的，而是根据状态自动重绘。
- `单一数据来源`：同一份数据最好只由一个地方真正拥有和维护。

## 一句话理解

谁拥有数据，谁定义 `@State`；谁只是帮忙修改数据，谁接收 `@Binding`。

## 先别急着背，先理解为什么会有这两个东西

你可以先想一个最简单的需求：

- 页面上有个数字
- 点按钮后数字加一

如果这是一个单独页面，最自然的做法就是：

- 这个页面自己存数字
- 这个页面自己改数字

这就是 `@State` 最适合的场景。

但如果按钮被你拆成了子视图呢？

- 数字还是父视图拥有
- 但按钮在子视图里
- 子视图也需要改这个数字

这时子视图就不应该重新拥有一份数字，而应该拿到父视图那份数字的“可修改引用”，这就是 `@Binding`。

## 手把手操作步骤

1. 创建父视图和子视图。
2. 粘贴代码并运行。
3. 点击按钮，观察父视图中的数字变化。
4. 把按钮文字改成“增加”，确认逻辑不受影响。
5. 试着再加一个“减 1”按钮，体会状态共用的感觉。

## 完整示例代码

```swift
import SwiftUI

struct CounterParentView: View {
    @State private var count = 0

    var body: some View {
        VStack(spacing: 12) {
            Text("当前值：\(count)")
                .font(.title3)

            CounterButtonView(count: $count)
        }
        .padding()
    }
}

struct CounterButtonView: View {
    @Binding var count: Int

    var body: some View {
        Button("加 1") {
            count += 1
        }
        .buttonStyle(.borderedProminent)
    }
}

#Preview {
    CounterParentView()
}
```

## 代码拆解（小白重点）

### `@State private var count = 0`

这表示：

- `count` 是会变化的数据
- 它属于 `CounterParentView`
- 只要它变化了，父视图就会重新计算界面

你可以先把 `@State` 理解成：这个页面自己的“内部状态记忆区”。

### `Text("当前值：\(count)")`

这里的界面内容直接依赖 `count`。所以只要 `count` 变化，显示出来的数字就会跟着变。

这就是“状态驱动界面”的最直观体现。

### `CounterButtonView(count: $count)`

这里最容易卡住的是 `$count`。

要先记住：

- `count` 是值
- `$count` 是绑定

也就是说：

- 如果你只是想读这个数字，通常用 `count`
- 如果你想把这份状态交给别人改，通常用 `$count`

### `@Binding var count: Int`

子视图这里不是重新创建状态，而是在说：

- 我这里有一份可以改的数字
- 但它真正属于别的视图

这就像你拿到了一把“可写权限”的钥匙，而不是把整套房子搬了一份进来。

### 点击按钮后为什么界面会更新

因为点击按钮后执行了：

```swift
count += 1
```

而这个 `count` 对应的是父视图的状态。状态变了，SwiftUI 就会自动重新计算界面，所以数字会立刻更新。

## 一个非常关键的判断题

以后你遇到状态问题时，可以先问自己：

### 问题 1：这份数据到底属于谁？

如果它属于当前页面自己，就优先考虑 `@State`。

### 问题 2：这个子视图是不是只负责帮忙修改？

如果是，就优先考虑 `@Binding`。

### 问题 3：是不是我不小心让父子各自维护了一份同样的数据？

如果是，就很容易出现状态不同步。

## 新手最容易混淆的地方

### 混淆 1：`@State` 和普通变量有什么区别

普通变量改了，SwiftUI 不一定知道界面要刷新。

`@State` 的意义就在于：它会告诉 SwiftUI“这个值变化了，界面可能要更新”。

### 混淆 2：子视图为什么不能直接也写一个 `@State count`

因为那样就变成了两份数据：

- 父视图一份
- 子视图一份

看起来名字一样，但它们其实已经不是同一份状态了。

### 混淆 3：为什么有时候界面能显示，但就是不同步

通常就是因为：

- 你传了值，不是绑定
- 或者你在子视图里偷偷自己维护了另一份状态

## 新手排错流程

1. 按钮能点但数字不变时，先检查父视图是不是用了 `@State`。
2. 检查传值时是不是用了 `$count`。
3. 检查子视图是不是用 `@Binding` 接收。
4. 如果父子界面数字不一致，先检查是不是各自维护了一份同名状态。
5. 如果你搞不清谁拥有数据，就先回到一句话：谁拥有数据，谁定义状态。

## 章节小测（带答案）

### 题 1

`@State` 通常放在哪里？

参考答案：放在拥有这份数据的视图里。

### 题 2

什么时候要用 `@Binding`？

参考答案：当子视图需要修改父视图的数据时。

### 题 3

`count` 和 `$count` 的区别是什么？

参考答案：`count` 是值，`$count` 是绑定。

## 练习任务

- 基础练习：增加一个“减 1”按钮。
- 加强练习：把“加 1”和“减 1”都拆到子视图里。
- 挑战练习：增加一个“重置为 0”按钮，再思考这个按钮该放父视图还是子视图更合适。

## 学习顺序建议

如果这一章你觉得抽象，不要急着往后跳。你至少应该把下面这件事做顺：

- 自己写一个有 `@State` 的小页面
- 再把一个按钮拆出去改成 `@Binding`

能把这一步做顺，后面章节会舒服很多。

## 本章学习提示

状态这一章非常关键。你后面学表单、列表、动画、网络加载，都会不断用到状态驱动的思维。

## 本章小结

学完这一章后，你应该已经真正理解：SwiftUI 的界面不是手动刷新出来的，而是状态变化后自动更新出来的；而 `@State` 和 `@Binding` 的区别，本质上是“拥有数据”和“协作修改数据”的区别。
