# 23. 发布流程基础：从开发到上线要经过什么

## 学习目标

- 理解一个 App 从写完到上线的大致流程。
- 知道开发、测试、构建、提交审核之间的关系。
- 对发布流程建立基础认知，不再只盯着写页面。

## 场景引入

很多新手会误以为“代码写完 = 项目完成”。但真实开发里，写完功能通常只是中间阶段。要让用户真的下载到你的 App，你还要经过：

- 自测
- 打包构建
- 提交到 App Store Connect
- 审核
- 发布

这一章不是教你把每个后台页面都背下来，而是先让你知道整条链路是什么。

## 本章术语先看懂

- `Build`：把代码编译成可运行应用。
- `Archive`：生成可用于发布的构建包。
- `TestFlight`：给测试人员分发测试版本的渠道。
- `App Store Connect`：管理应用上架和版本信息的平台。

## 一句话理解

开发完成只是第一步，发布流程是在把“代码”变成“用户能下载的产品”。

## 一个基础发布流程

1. 开发完主要功能。
2. 自己先完整测试关键流程。
3. 修掉明显 bug 和崩溃问题。
4. 在 Xcode 中 Archive。
5. 上传到 App Store Connect。
6. 先走 TestFlight 内测。
7. 确认版本稳定后提交审核。
8. 审核通过后正式发布。

## 每一步在解决什么问题

### 自测

避免把明显问题直接带到测试或审核阶段。

### Archive

生成正式可分发的版本，而不是仅仅本地运行。

### TestFlight

让团队、朋友或测试人员先体验真实安装包。

### 审核

确认应用符合平台规则和基本质量要求。

## 新手常见误区

- 以为模拟器里能跑就代表可以发布。
- 没经过完整测试就急着提交审核。
- 只关注功能，不关注版本说明、截图、元数据。

## 新手排错流程

1. 发布前先列出关键用户路径，逐一手测。
2. Archive 失败时，先检查签名、配置和构建错误。
3. TestFlight 体验不对时，确认测试的是不是最新包。
4. 审核被拒时，先看清拒绝原因，再有针对性修复。

## 章节小测（带答案）

### 题 1

为什么代码写完不等于项目完成？

参考答案：因为还要经过测试、构建、上传、审核和发布等流程。

### 题 2

TestFlight 最常见的用途是什么？

参考答案：给测试人员和团队分发测试版本。

### 题 3

为什么发布前一定要先自测？

参考答案：因为很多明显问题在自测阶段就能提前发现，避免把低级错误带到后续流程。

## 练习任务

- 基础练习：为你的小项目列一个“发布前检查清单”。
- 加强练习：整理一份“关键用户流程”测试列表。
- 挑战练习：模拟写一份版本更新说明。

## 本章小结

学完这章后，你应该已经知道：发布不是最后点一下按钮，而是一整套把代码变成产品的流程。
