# 38. 推送通知基础：不是发出去就算完成

## 学习目标

- 理解推送通知的核心价值和常见使用场景。
- 知道什么样的通知更容易有价值，什么样的通知容易打扰用户。
- 建立对通知设计的基础认知。

## 场景引入

推送通知看起来像是“把消息发给用户”这么简单，但实际上如果设计不好，它很容易变成干扰：

- 频率太高
- 内容太空
- 时机不对
- 用户根本不关心

真正有价值的通知，应该帮助用户在正确时间注意到重要事情，而不是让用户烦。

## 本章术语先看懂

- `推送通知`：由系统发送给用户的消息提醒。
- `触达`：把信息送到用户面前。
- `时机`：通知出现的时间点。
- `相关性`：通知内容和用户当前需求的匹配程度。

## 一句话理解

好的通知不是“多发”，而是“在对的时间发对用户有用的内容”。

## 适合做通知的场景

- 学习提醒即将开始
- 某个任务今天还没完成
- 重要活动即将截止
- 用户主动订阅的更新内容

## 不适合乱发通知的场景

- 内容太普通，没有明确价值
- 过度促活，只是为了刷存在感
- 用户已经完成该任务，却还继续提醒

## 新手常见误区

- 把通知当成万能召回手段。
- 只想发出去，不关心用户为什么要点开。
- 通知文案太空，看不出价值。

## 新手排错流程

1. 每条通知先问自己：用户为什么值得被打扰？
2. 先从少量高价值通知开始，不要一上来做很多类。
3. 设计通知时同时考虑“什么时候发”和“点开后去哪”。

## 章节小测（带答案）

### 题 1

为什么通知不是发出去就算完成？

参考答案：因为真正的价值在于用户是否愿意看、是否觉得有用、是否愿意继续行动。

### 题 2

什么样的通知更容易打扰用户？

参考答案：频率高、时机不对、内容空洞或与用户无关的通知。

### 题 3

设计通知时为什么要考虑“点开后去哪”？

参考答案：因为通知只是入口，后续落点体验会直接影响用户是否感到顺畅。

## 练习任务

- 基础练习：为你的应用写 3 条真正有价值的通知文案。
- 加强练习：给每条通知标出最合适的发送时机。
- 挑战练习：为通知点击后的页面落点设计一条更顺畅的用户路径。

## 本章小结

学完这章后，你应该已经知道：推送通知的重点不在“能不能发”，而在“发了以后对用户有没有价值”。
