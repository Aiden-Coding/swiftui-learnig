# 33. 本地化基础：让应用能支持多语言

## 学习目标

- 理解为什么文案不应该全部写死在界面里。
- 知道本地化最基础的工作方式。
- 建立对多语言支持的基本认知。

## 场景引入

如果你的 App 只给自己用，中文写死在页面里问题也不大。但只要你希望它支持更多用户，很快就会遇到：

- 不同语言的界面文案需求
- 同一按钮在不同语言下长度不同
- 页面布局被更长的文字撑开

本地化不是简单翻译，而是让应用从结构上支持多语言切换。

## 本章术语先看懂

- `本地化`：根据不同语言和地区显示不同文案和格式。
- `LocalizedStringKey`：SwiftUI 中常见的本地化字符串键用法。
- `多语言资源`：不同语言对应的文案内容。
- `文案键`：用于查找不同语言文本的标识。

## 一句话理解

支持多语言的第一步，不是先翻译，而是别把所有文案都直接写死在界面逻辑里。

## 基础示例

```swift
import SwiftUI

struct LocalizationDemoView: View {
    var body: some View {
        VStack(spacing: 12) {
            Text("home_title")
                .font(.title2)
                .fontWeight(.bold)

            Text("home_subtitle")
                .foregroundStyle(.secondary)

            Button("start_learning") {
                print("开始学习")
            }
            .buttonStyle(.borderedProminent)
        }
        .padding()
    }
}

#Preview {
    LocalizationDemoView()
}
```

## 代码拆解（小白重点）

- 这里的 `"home_title"` 不再是最终展示文案，而是一个文案键。
- 不同语言下，这个键可以映射成不同内容。
- 这样以后换语言时，不需要逐个页面改代码。

## 为什么本地化不只是翻译

因为除了文字内容本身，还会影响：

- 按钮长度
- 排版空间
- 日期、时间、数字格式
- 用户对文案语气的理解

所以本地化也是界面设计和工程组织的一部分。

## 新手常见误区

- 以为后面统一查找替换就能轻松本地化。
- 所有文案都直接写死在页面结构里。
- 只考虑文案内容，不考虑不同语言的长度变化。

## 新手排错流程

1. 先检查项目里哪些页面文案还全是硬编码。
2. 为高频、核心页面优先提取文案键。
3. 检查按钮和标题在更长文案下会不会布局变形。

## 章节小测（带答案）

### 题 1

为什么不建议把所有界面文案都直接写死？

参考答案：因为后续支持多语言会非常难改，也不利于统一管理。

### 题 2

本地化除了翻译文本，还会影响什么？

参考答案：还会影响布局、日期格式、长度适配和整体界面体验。

### 题 3

本地化的第一步最重要的是什么？

参考答案：先建立文案可配置、可替换的结构，而不是全部写死在页面里。

## 练习任务

- 基础练习：把一个页面里的 5 条文案都提成键。
- 加强练习：尝试为中英文分别设计更自然的按钮文字。
- 挑战练习：检查一个页面在更长英文文案下会不会排版变形。

## 本章小结

学完这章后，你应该已经知道：本地化的关键不是“翻译得快”，而是从一开始就把文案组织方式设计对。
