# 44. 模块化基础：项目变大后，边界要更清楚

## 学习目标

- 理解为什么项目变大后要考虑模块边界。
- 知道模块化和单文件拆分不是一回事。
- 建立对“按职责分组、减少耦合”的基础认知。

## 场景引入

当项目还小时，所有代码都在一个 target 或几个文件夹里，通常还勉强能管理。但随着功能越来越多，就容易出现：

- 什么代码都能随便互相调用
- 改一个页面牵扯很多地方
- 业务边界越来越模糊
- 编译和协作都越来越痛苦

模块化的意义，就是让项目结构更有边界感，而不是所有东西都混在一起。

## 本章术语先看懂

- `模块化`：把项目按职责拆成更清晰的部分。
- `耦合`：模块之间互相依赖过深，改一个就容易影响另一个。
- `边界`：一个模块应该负责什么、不应该负责什么。
- `职责划分`：根据功能和层次分配代码归属。

## 一句话理解

模块化不是把目录拆得更碎，而是让不同职责之间更少互相缠绕。

## 什么时候该开始有模块化意识

- 页面和功能明显分成多个业务区。
- 团队协作开始增多。
- 代码互相引用越来越杂。
- 新人很难快速看懂结构。

## 新手常见误区

- 以为多建几个文件夹就等于模块化。
- 为了模块化而模块化，结果结构更复杂。
- 边界没定清楚就硬拆，最后还是互相引用一团乱。

## 新手排错流程

1. 先盘点项目里有哪些清晰的业务边界。
2. 从最明显的一块职责开始拆，而不是一口气全面重构。
3. 模块化后要继续检查依赖方向是否合理，避免只是表面拆开。

## 章节小测（带答案）

### 题 1

模块化最核心的目标是什么？

参考答案：让项目结构边界更清晰、耦合更低、维护更容易。

### 题 2

为什么“多几个文件夹”不等于模块化？

参考答案：因为真正的模块化关注的是职责边界和依赖关系，不只是目录数量。

### 题 3

什么时候值得开始有模块化意识？

参考答案：当功能增多、边界变复杂、协作变多、互相依赖混乱时。

## 练习任务

- 基础练习：为你的项目画一个简单的职责分区图。
- 加强练习：找出一个目前耦合最重的区域，写出拆分思路。
- 挑战练习：把“页面层、业务层、数据层”的边界再按模块角度重想一遍。

## 本章小结

学完这章后，你应该已经知道：模块化的重点不是拆得多，而是让项目边界更清楚、依赖更健康。
